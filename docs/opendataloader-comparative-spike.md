# OpenDataLoader Comparative Spike

Date: 2026-05-04

## Scope

This spike compares OpenDataLoader PDF's public data model and runtime shape against LarkScout's current scanned-contract/table pipeline. It does not add OpenDataLoader, Java, or any new required runtime dependency to LarkScout.

OpenDataLoader runtime execution was not completed in this environment:

- `java -version` failed with no Java Runtime available.
- `opendataloader-pdf` is not installed in the active Python environment.
- The OpenDataLoader README states Java 11+ is required for the Python package path.

The comparison below therefore uses:

- OpenDataLoader public README, `schema.json`, and `options.json`.
- Three local LarkScout PDF samples from `/Users/grace/.larkscout/docs`.
- Existing LarkScout outputs already present in the local document library.

## OpenDataLoader Observations

OpenDataLoader's advertised output shape is element-centric:

- Top-level JSON has file metadata and a `kids` array.
- Each content element has `type`, `page number`, and `bounding box`.
- Bounding boxes use `[left, bottom, right, top]` in PDF points.
- Tables are nested semantic objects with `number of rows`, `number of columns`, `rows`, and table-cell `kids`.
- Cross-page table/list continuity is modeled with `previous table id`, `next table id`, `previous list id`, and `next list id`.
- CLI options expose `reading-order`, `table-method`, `pages`, `image-output`, `use-struct-tree`, and content-safety controls.

OpenDataLoader runtime/operational constraints:

- The Python quick start still requires Java 11+.
- Repeated `convert()` calls spawn JVM work, so batching is recommended.
- OCR/scanned workflows are tied to hybrid mode and a separate local backend.

## LarkScout Comparison

| Area | OpenDataLoader model | Current LarkScout state | Reuse decision |
| --- | --- | --- | --- |
| Geometry | Element-level bbox in PDF points | OCR blocks and reconstructed tables use `image_pixels` with source page dimensions | Add explicit transform metadata before changing coordinate defaults |
| Structure | Unified `kids` tree across headings, paragraphs, tables, lists, images | Sections, tables, images, OCR blocks are separate sidecars | Consider optional `elements.json`; do not collapse existing APIs |
| Tables | Nested rows/cells with bbox, spans, continuity ids | Structured table JSON has rows/cells, bbox, spans, OCR refs, `continued_from/to` | Already aligned; add stronger cell nesting only when needed |
| Discovery | JSON/Markdown/HTML outputs generated by explicit format | Manifest stays low-token; P1.7 adds explicit sidecar discovery and targeted page/table calls | Keep bounded discovery model |
| Visual debug | Annotated output is a first-class output format | P1.6 adds opt-in derived debug overlays | Keep as derived artifact, not default output |
| OCR | Hybrid OCR mode with language/backend options | Local PaddleOCR/LLM OCR paths plus region rerun helper | Do not import hybrid runtime; keep backend-neutral helpers |
| Safety | Hidden/off-page/tiny text filters and sanitization options | LarkScout has SSRF/path controls, but no PDF hidden-text safety sidecar yet | Reusable idea for future OCR/layout quality flags |

## Reusable Ideas

1. Add an optional element-tree sidecar later.
   LarkScout should keep section/table APIs stable, but an `elements.json` sidecar could expose semantic items with `id`, `type`, `page`, `bbox`, `children`, and `source_refs`.

2. Preserve coordinate-system metadata everywhere.
   OpenDataLoader's PDF-point bbox model is easier for PDF citation UX. LarkScout should keep `image_pixels` for OCR evidence but record page pixel size, PDF point size, and conversion transforms in sidecars.

3. Extend safety metadata.
   OpenDataLoader's hidden/off-page/tiny text controls suggest future flags on OCR blocks or text spans: `hidden_like`, `off_page_like`, `tiny_text_like`, and `watermark_like`.

4. Keep visual verification cheap and explicit.
   P1.6's `derived/debug/` path matches the useful part of OpenDataLoader's annotated output without forcing debug assets into default responses.

5. Treat cross-page continuity as metadata, not merged content.
   P1.3's `continued_from/to` mirrors OpenDataLoader's previous/next ids while preserving separate page/table artifacts.

## Non-Reusable Choices

- Do not add Java as a LarkScout requirement.
- Do not wrap OpenDataLoader as a production backend until there is a dedicated dependency and deployment decision.
- Do not adopt OpenDataLoader's hybrid server model for core parsing; LarkScout's region crop and rerun helpers are narrower and easier to operate.
- Do not switch default bbox conventions until downstream consumers can handle coordinate transforms.
- PDF tagging/PDF-UA output is out of scope for the scanned contract/invoice table plan.
